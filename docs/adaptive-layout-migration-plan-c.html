1. 迁移目标与原则

1.1 业务目标

  • 完全对齐 Material 3 自适应规范（5 档断点 + 3 类 canonical layouts）；
  • 消除当前因 600 / 1200 双阈值导致的"组件错位"现象；
  • 把"列表→详情"、"主区+辅助区"、"内容流"这三种交互让框架托管，业务代码不再手写双栏/三栏 Row；
  • WasmJS / Android 行为完全一致，保持 KMP commonMain 单点实现。

1.2 工程原则

  • 分阶段、可回滚：每阶段都能独立合并、独立验证，必要时单独 revert；
  • 共存优先：旧 WindowSizeClass/ListDetailLayout 与新 scaffold 并存到阶段 8 才统一删除，避免一次性大改；
  • 不引入新业务逻辑：迁移只动布局层，不动 ViewModel/Repository；
  • Experimental API 集中管理：所有 @ExperimentalMaterial3AdaptiveApi opt-in 通过 module 级 freeCompilerArgs += "-opt-in=..." 统一声明，业务文件不写 opt-in 注解。

2.2 导航 Shell 架构

顶层 Shell 用官方 NavigationSuiteScaffold（M3 stable，CMP 已通过 compose.material3AdaptiveNavigationSuite 接入）， 替代当前手写的 CompactNavigationLayout / MediumNavigationLayout / RailNavigationLayout。 scaffold 内部仍承载 NavDisplay（nav3），canonical pane 出现在每个目的地的实现里。 

NavigationSuiteScaffold(
  // 框架托管 NavBar / NavRail / NavDrawer 切换
  layoutType = NavigationSuiteScaffoldDefaults.calculateFromAdaptiveInfo(currentWindowAdaptiveInfo()),
) {
  // 内部仍是 nav3 的 NavDisplay；AgentSidebar / Toast 等全局层挂这里
  NavDisplay(entries = ..., sceneStrategies = listOf(rememberListDetailSceneStrategy()))
}

2.3 关键不动点（守护性约束）

  • 所有页面入口仍是 NavDest.*，nav3 backstack 不动；
  • AgentSidebar / AgentFloatingCard 行为不变，只是挂载点从手写三栏挪到 NavigationSuiteScaffold 的 content slot；
  • ViewModel 契约（Contract.State / Action / Effect）一字不动；
  • ProvideMobileTitleBar 在 Compact 下继续可用（Scaffold topBar 不被 canonical scaffold 接管）。

6. 阶段 1：WindowSizeClass 重写

6.1 设计

新 ui/core/adaptive/m3/AdaptiveInfo.kt 完全用官方 currentWindowAdaptiveInfo()， 对外暴露和现有 AdaptiveLayoutConfig 语义兼容的接口，使老页面无须立刻改。 

@Stable
data class AdaptiveLayoutConfig(
  val widthSizeClass: WindowWidthSizeClass,
  val heightSizeClass: WindowHeightSizeClass,
  val posture: Posture,
) {
  val isCompact:  Boolean get() = widthSizeClass == WindowWidthSizeClass.COMPACT
  val isMedium:   Boolean get() = widthSizeClass == WindowWidthSizeClass.MEDIUM   // 600..839
  val isExpanded: Boolean get() = widthSizeClass == WindowWidthSizeClass.EXPANDED // 840..1199
  val isLarge:    Boolean get() = widthSizeClass == WindowWidthSizeClass.LARGE    // 1200..1599
  val isXLarge:   Boolean get() = widthSizeClass == WindowWidthSizeClass.EXTRA_LARGE // >=1600

  /** 兼容旧语义：Compact 单栏；其余双栏 */
  val useTwoPane: Boolean get() = !isCompact

  /** 兼容旧语义：底部导航仅 Compact */
  val useBottomNavigation: Boolean get() = isCompact
  val useNavigationRail:   Boolean get() = isMedium || isExpanded
  val useNavigationDrawer: Boolean get() = isLarge  || isXLarge
}

@Composable
fun rememberAdaptiveLayoutConfig(): AdaptiveLayoutConfig {
  val info = currentWindowAdaptiveInfo(supportLargeAndXLargeWidth = true)
  return remember(info) {
    AdaptiveLayoutConfig(info.windowSizeClass.windowWidthSizeClass,
                        info.windowSizeClass.windowHeightSizeClass,
                        info.windowPosture)
  }
}

6.2 兼容层

  • 旧 org.shiroumi.quant_kmp.ui.core.adaptive.AdaptiveLayoutConfig 改为 typealias 指向新类型；
  • 旧 rememberAdaptiveLayoutConfig() 函数体改为转发到新实现，不删函数签名；
  • WindowSizeClass.kt 中的 sealed class 三档保留，但每个 data object 多加一个 fromM3() 工厂，把 LARGE/XL 折叠到 Expanded（语义降级），保证现有所有页面分支不爆。

6.3 任务

7.1 目标

把 Navigation.kt 中手写的三个 NavigationLayout（Compact / Medium / Rail）合并成单一 NavigationSuiteScaffold，由 M3 自动决定 NavBar / NavRail / NavDrawer。

7.2 改造前 vs 改造后

// 改造前

  config.isCompact -> CompactNavigationLayout(...)
  config.isMedium  -> MediumNavigationLayout(...)

// 改造后
NavigationSuiteScaffold(

8. 阶段 3：Candle 页面迁移到 ListDetailPaneScaffold

8.1 现状分析

当前 CandleScreen.kt 在 Compact 下走"CandleChartSection 全屏 + 顶部按钮拉出 StockListBottomSheet"， 在 Medium/Expanded 下走"Row(StockListPanel, CandleChartSection)"。列表选中态由 CandleViewModel 维护。

8.2 目标结构

val navigator = rememberListDetailPaneScaffoldNavigator<String>() // contentKey = tsCode

NavigableListDetailPaneScaffold(
  navigator = navigator,
  listPane  = {
    AnimatedPane {
      StockListPanel(
        stocks         = state.stocks,
        selectedTsCode = navigator.currentDestination?.contentKey,
        onSelect       = { tsCode ->
          viewModel.dispatch(SelectStock(tsCode))
          scope.launch { navigator.navigateTo(ListDetailPaneScaffoldRole.Detail, tsCode) }
        },
      )
    }
  },
  detailPane = {
    AnimatedPane {
      val tsCode = navigator.currentDestination?.contentKey
      if (tsCode != null) {
        CandleChartSection(
          state     = state,
          tsCode    = tsCode,
          isCompact = scaffoldValueIsSinglePane(navigator),
          onBack    = if (scaffoldValueIsSinglePane(navigator))
                        { { scope.launch { navigator.navigateBack() } } }
                      else null,
        )
      } else {
        EmptyDetailHint()
      }
    }
  },
)

8.3 处理要点

  • navigator 与 ViewModel 的双向同步： 选中态依然以 ViewModel 为准（涉及 WebSocket 订阅切换），navigator 只承担 UI 姿态。navigateTo 是触发 UI 切换，dispatch(SelectStock) 才触发数据订阅。
  • Compact 不再用 BottomSheet：scaffold 已经在 Compact 下自动单栏切换； 可去掉 StockListBottomSheet 与 isStockListExpanded 状态。如果产品强烈希望保留"列表抽屉"手感， 可以在 listPane 内裹一个全屏 Container，但更建议遵循 canonical。
  • 顶部栏：Compact 下用 ProvideMobileTitleBar，返回箭头绑定到 navigator.navigateBack()；Medium+ 不显示返回箭头。 需要写一个 helper fun ThreePaneScaffoldNavigator<*>.isCurrentlySinglePane(): Boolean 判断单栏态。
  • 列表与详情共享 state：listPane 里读 state.stocks，detailPane 里读 state.candleData。 不要把整个 CandleScreen 拆分到两个 file，保持单文件管理 contract。
  • K 线指标 BottomSheet 不变，仍由 CandleChartSection 内部处理。

8.4 任务

  • ui/core/adaptive/m3/PaneHelpers.kt 提供 isCurrentlySinglePane、animatedPaneOrNull；
  • CandleScreen.kt 整体重写为单一 NavigableListDetailPaneScaffold，移除 StockListBottomSheet；
  • 测试：
    • Wasm 440dp：仅列表，点项进详情，返回箭头回到列表；
    • Wasm 800dp（M3 Medium，单栏）：与 440dp 行为一致；
    • Wasm 1000dp（M3 Expanded，双栏）：列表 + 详情并列，点项无返回；
    • Wasm 1500dp（M3 Large）：相同双栏，scaffold 自动扩列表宽度上限；
    • Android 旋转横竖各档自动切换；
  • 验证 WebSocket 订阅切换在 navigator 切详情时不掉链；
  • 更新 skill 文档 §6.2。

10. 阶段 5：Agent 独立工作区方案撤销

10.1 现状难点

原 Agent 独立工作区方案曾计划使用 Compact 单列、Medium 双列、Expanded 三列， 且每个分隔条都用 DraggableDivider 手动维护宽度。这不是 ListDetail 的形态（list 不是数据选择），更像 "main = chat" + 两个 supporting（左：会话历史；右：详情）。

10.2 方案选择

方案组件优点缺点
A. 单 ListDetail，三 pane ListDetail + extraPane 原生支持 list/detail/extra 语义错位（chat 是 detail？session 是 list？extra 又是详情？三者优先级不对）
B. 单 SupportingPane，左侧 supporting 折叠 SupportingPaneScaffold main = chat 语义干净 无法同时挂两个 supporting
C. SupportingPane 嵌套 SupportingPane（推荐） 外层 main=chat / supporting=details；内层 main=外层全部 / supporting=sessionList 语义正确；两个 supporting 各自按断点折叠；可保留 Drag 行为 两个 navigator，需要状态同步

10.3 当前结论

独立 Agent 工作区方案已撤销，不再作为顶级导航入口；当前保留 Sidebar/FloatingCard 快捷入口承载对话、工具审批和上下文分析。

10.4 要点

  • 放弃 DraggableDivider 的拖拽宽度调节（M3 1.2+ 才有官方 paneExpansionDraggable，且仍 Experimental）。 如果业务上拖拽是刚需，单独立项；本计划默认不保留拖拽；
  • 不再维护页面级 session/detail supporting pane；相关能力收敛到 Sidebar/FloatingCard。

10.5 任务

  • 删除独立 Agent 工作区顶级入口和页面专属 Composable；
  • 保留 Sidebar/FloatingCard 对话入口；
  • 更新 skill 文档，避免后续按旧方案恢复顶级 tab。

14.2 回滚策略

  • 每阶段独立 PR，独立 revert；
  • 阶段 1（兼容层）一旦合并，所有页面仍能跑老代码，这是一个稳定锚点；
  • 阶段 3/4/5 任一页面出问题，单独 revert 该页面 PR，不影响其他页面与 shell；
  • 阶段 8 之前，旧 API 都不删，回滚成本接近零。
