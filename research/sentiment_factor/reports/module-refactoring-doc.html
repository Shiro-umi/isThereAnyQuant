频域共振研究管线 · 模块化重构

将原有"研究管线直接读 DB"的单体架构重构为五个可复用的独立模块， 通过统一的 FactorDataSource 接口实现研究环境（DB）和生产环境（内存快照）的完全一致的代码路径。 重构后研究输出 611 qualified A 卡片，与重构前完全一致。

§2 模块架构图

┌──────────────────────────────────────────────────────────────────┐ │ shared (contract) │ │ FactorSnapshot │ MarketRegime │ SelectionRule │ QualifiedResonance│ │ FactorDataSource (interface) │ └──────┬───────────────┬────────────────────┬───────────────────────┘ │ │ │ ▼ ▼ ▼ ┌──────────────┐ ┌──────────────┐ ┌──────────────────────────┐ │ M1 research │ │ M1 service │ │ M5 core │ │ DbFactorData │ │ SnapshotFact │ │ SelectionRuleEngine │ │ Source │ │ orDataSource │ │ │ │ │ │ │ │ load(cards) │ │ DB→Snapshot │ │ ConcurrentHM │ │ forRegime(regime)→[Rule] │ └──┬───────────┘ └──────┬───────┘ └──────────────────────────┘ │ │ │ ┌─────────────────┘ │ │ 共同实现 FactorDataSource 接口 ▼ ▼ ┌──────────────────────────────────────────────────────────────────┐ │ M3 research: SentimentResonanceStudy │ │ │ │ constructor(dataSource: FactorDataSource) ← 注入接口，不感知实现│ │ run(ctx, Unit) → dataSource.history() → toRecord() → runRecords()│ │ runRecords(ctx, records) → List<ResonanceMetric> ← 核心算法零改动│ └───────────────────────────┬──────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────────┐ │ M4 research: ResonanceEvaluator │ │ evaluate(Metric) → Report → Card (现有代码, 零改动)│ └───────────────────────────┬──────────────────────────────────────┘ │ ┌─────────────┴─────────────┐ ▼ ▼ ┌──────────────────┐ ┌──────────────────────────┐ │ M2 core │ │ M5 core │ │ StateClassifier │ │ SelectionRuleEngine │ │ │ │ │ │ classify(snap, │ │ load(cards) │ │ history)→Regime│ │ forRegime(regime)→[Rule] │ └──────────────────┘ └──────────────────────────┘
关键设计原则： ① 研究代码即生产代码（同一份 SentimentResonanceStudy.runRecords() 和 ResonanceEvaluator.evaluate()）。
② 唯一变化点是 FactorDataSource 的实现——研究用 DB，生产用内存快照。两者实现同一接口，Study 不感知差异。
③ StateClassifier 和 SelectionRuleEngine 是纯函数/纯内存模块，无 IO，可被任意环境高频调用。
④ research 模块依然不依赖 core/service（遵守原有边界约束）。

§4 生产环境用法（盘后 + 盘中）

4.1 应用启动时：一次性加载共振卡片

import org.shiroumi.strategy.core.resonance.SelectionRuleEngine
import kotlinx.serialization.json.Json

// 从研究产出的 JSON 文件加载 qualified 卡片
val json = Json { ignoreUnknownKeys = true }
val cards = json.decodeFromString<List<QualifiedResonance>>(
    File("research/sentiment_factor/out/resonance_cards/*.json").readText()
)
SelectionRuleEngine.load(cards.filter { it.qualified })
// 之后所有查询为 O(1) HashMap 查找

4.2 盘后（15:05）：刷新快照 + 状态分类 + 生成规则

val snapshotSource = SnapshotFactorDataSource()

// Step 1: 盘后因子落库后，全量刷新内存快照
snapshotSource.refresh(startDate = today.minusDays(600), endDate = today)

// Step 2: 获取当日因子值 + 历史窗口
val todaySnapshot = snapshotSource.snapshot(today)!!
val history = snapshotSource.history(today.minusDays(200), today)

// Step 3: 分类当前市场状态
val regime = StateClassifier.classify(todaySnapshot, history)!!
println("今日状态: trend=${regime.trendLevel}, disp=${regime.dispersionLevel}, vol=${regime.volumeLevel}")
println("业务分类: ${regime.regimeCategory}")
println("stateId: ${regime.stateId}")

// Step 4: 查询适用选股规则
val rules = SelectionRuleEngine.forRegime(regime)
rules.forEach { rule ->
    println("${rule.name}: 持仓${rule.horizon}天, 仓位${(rule.positionRatio*100).toInt()}%, 预计${rule.estimatedPoolSize}只候选")
    rule.filters.forEach { f ->
        println("  ${f.dimension} ${f.operator} ${f.threshold}  # ${f.description}")
    }
}

4.3 盘中（9:30-15:00）：高频查询

// 盘中任意时刻触发（如价格更新、定时轮询）
suspend fun onIntradayRefresh() {
    // 内存读取，无 IO，无锁，< 1μs
    val todaySnapshot = snapshotSource.snapshot(today) ?: return
    val history = snapshotSource.history(today.minusDays(200), today)

    // 纯计算，< 10μs
    val regime = StateClassifier.classify(todaySnapshot, history) ?: return

    // HashMap 查找，< 1μs
    val rules = SelectionRuleEngine.forRegime(regime)

    // 套用规则到实时行情，生成盘中调仓建议
    val candidates = applyRulesToRealtimePrices(rules, realtimeQuotes)
    pushToWebSocket(candidates)
}

4.4 盘后增量追加（轻量模式）

// 每天只需追加今天的快照，不需要全量 refresh
val todayRecord = /* 当日新计算的因子记录 */
val todaySnapshot = todayRecord.toSnapshot()
snapshotSource.append(todaySnapshot)  // O(log N)，增量追加

§5 环境切换指南

场景FactorDataSourceStudy 构造StateClassifierSelectionRuleEngine
研究迭代 DbFactorDataSource SentimentResonanceStudy(DbFactorDataSource()) 不使用 不使用（研究走 Evaluator→Card 路径）
回测验证 DbFactorDataSource SentimentResonanceStudy(DbFactorDataSource()) StateClassifier.classify(snapshot, history) SelectionRuleEngine.load(cards) 然后 forRegime()
盘后生产 SnapshotFactorDataSource 不使用 StateClassifier.classify(snapshot, history) SelectionRuleEngine.forRegime(regime)
盘中高频 SnapshotFactorDataSource 不使用 StateClassifier.classify(snapshot, history) SelectionRuleEngine.forRegime(regime)
切换要点： ① 研究和生产共享 FactorDataSource 接口——换一个实现就换了环境。
② 研究环境不需要 StateClassifier 和 SelectionRuleEngine（走 Evaluator → JSON → 分析报告路径）。
③ 生产环境不需要 SentimentResonanceStudy（共振卡片已由研究预计算，加载 JSON 即可）。
④ 唯一同时被两边调用的模块是 FactorDataSource 接口和共享数据契约（FactorSnapshot 等）。

§6 数据契约详解

6.1 FactorSnapshot（因子快照）

研究侧和生产侧之间唯一的数据交换格式。与 SentimentFactorDailyRecord 字段一一对应，往返无损。

字段类型说明
tradeDateLocalDate交易日
factorsMap<String, Double?>38 个因子名 → 取值（可含 null）
y1Raw / y2Raw / y3RawDouble?三个 Y 分量的 T 日原始值
yCompositeDouble?综合情绪指标
notesString?备注（极端期标记等）

6.2 FactorDataSource（因子数据源接口）

方法返回研究环境实现生产环境实现
snapshot(date)FactorSnapshot?DB 单日查询ConcurrentHashMap.get()，O(1)，无锁
history(start, end)List<FactorSnapshot>DB 区间查询 + 排序二分查找 + 遍历，O(log N + K)
latestTradeDate()LocalDate?从今天往前找有数据的日期CopyOnWriteArrayList.last()，O(1)

6.3 MarketRegime（市场状态）

字段类型说明
tradeDateLocalDate交易日
trendLevel / dispersionLevel / volumeLevelTrendLevelLOW / MID / HIGH
regimeCategoryRegimeCategory六大业务分类之一
stateIdString格式 "trend=low,disp=mid,vol=high"，与 ResonanceCard.state_id 对齐

6.4 SelectionRule（选股规则）

字段类型说明
nameString规则名称，如"小盘领涨""广度补涨"
regimeStateIdString适用状态标识
evidenceCardCountInt支撑此规则的 A 类共振卡片数
targetYStringY1 / Y2 / Y3
horizonInt建议持仓天数 1/3/5
positionRatioDouble建议仓位 0-1（基于 mean_coherence 自动计算）
filtersList<StockFilter>具体筛选条件
exitConditionString出场条件描述
dominantBandString主导频带 F1b/F2a
estimatedPoolSizeInt预估候选股票数
